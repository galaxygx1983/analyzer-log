# 边缘侧分布式日志管理系统设想

## 基于 Redis + Pino + 大模型 + PostgreSQL 的轻量级技术架构

---

## 一、系统概述

### 1.1 设计原则：极简轻量

本方案遵循**极简架构**原则，仅保留核心组件：

| 组件                      | 职责                        | 理由                                 |
| ------------------------- | --------------------------- | ------------------------------------ |
| **Go应用**          | 日志采集 + 业务逻辑         | 原生高性能，内存占用低               |
| **file-collector**  | 第三方日志文件采集          | 无侵入式采集第三方程序日志           |
| **Redis**           | 日志存储 + 消息队列         | 单组件解决存储和传输                 |
| **Pino**            | 日志管理 (查询/过滤/格式化) | 超高性能Node.js日志库，比Bunyan快5倍 |
| **大模型（skill）** | 智能分析                    | 异常检测、根因分析                   |
| **PostgreSQL**      | 分析结果持久化              | 关系型数据库，支持复杂查询           |

### 1.3 日志库选型对比

| 特性       | Pino         | Bunyan     | Winston      |
| ---------- | ------------ | ---------- | ------------ |
| 性能       | ~5倍于Bunyan | 基准       | 较慢 (~10倍) |
| 依赖       | 极少         | 较多       | 较多         |
| 输出格式   | JSON         | JSON       | 多格式       |
| 子日志器   | 支持         | 支持       | 支持         |
| 格式化工具 | pino-pretty  | bunyan CLI | 内置         |
| 浏览器支持 | 支持         | 不支持     | 支持         |
| 传输层     | 需自行实现   | 需自行实现 | 内置多传输   |

**选择 Pino 的原因：**

1. **极致性能**：边缘设备资源有限，Pino的高性能可降低CPU和内存占用
2. **轻量依赖**：依赖极少，适合边缘侧部署
3. **Bunyan兼容**：官方认证的Bunyan现代化版本，可直接使用bunyan CLI工具
4. **灵活扩展**：核心简洁，可按需添加传输层，避免Winston的冗余开销

### 2.2 数据存储策略

根据业务需求和资源条件，提供多种存储策略供选择：

#### 策略一：日志分级存储（推荐）

| 日志级别         | Redis存储     | 本地文件      | 适用场景             |
| ---------------- | ------------- | ------------- | -------------------- |
| DEBUG / TRACE    | ✓ (2天有效)  | ✗            | 开发调试，短期保留   |
| INFO             | ✓ (7天有效)  | ✗            | 常规运行日志         |
| WARNING          | ✓ (15天有效) | ✓ (轮转保留) | 需要关注的异常       |
| ERROR / CRITICAL | ✓ (15天有效) | ✓ (轮转保留) | 严重错误，需长期追溯 |

**优点**：平衡存储成本与追溯能力，适合生产环境

#### 策略二：纯Redis存储

| 日志级别 | Redis存储       | 本地文件 |
| -------- | --------------- | -------- |
| 全部级别 | ✓ (有效期不同) | ✗       |

**优点**：架构最简单，无本地文件管理开销
**缺点**：Redis故障可能丢失日志，依赖Redis持久化

#### 策略三：双写同步

| 日志级别 | Redis存储 | 本地文件      |
| -------- | --------- | ------------- |
| 全部级别 | ✓        | ✓ (同步写入) |

**优点**：最高可靠性，双重保障
**缺点**：写入性能受本地IO影响

#### 策略四：本地优先 + 异常上传

| 日志级别                   | Redis存储 | 本地文件 |
| -------------------------- | --------- | -------- |
| DEBUG / TRACE / INFO       | ✗        | ✓       |
| WARNING / ERROR / CRITICAL | ✓        | ✓       |

**优点**：最小化网络传输，适合网络不稳定的边缘环境
**缺点**：低级别日志无法实时分析

#### 策略五：采样存储

| 日志级别         | Redis存储 | 本地文件  | 采样率 |
| ---------------- | --------- | --------- | ------ |
| DEBUG / TRACE    | ✓ (采样) | ✗        | 1%     |
| INFO             | ✓ (采样) | ✗        | 10%    |
| WARNING          | ✓ (全量) | ✓ (轮转) | 100%   |
| ERROR / CRITICAL | ✓ (全量) | ✓ (轮转) | 100%   |

**优点**：大幅降低存储成本，保留关键日志完整性
**缺点**：低级别日志可能遗漏问题细节
**适用场景**：日志量巨大、存储成本敏感

#### 策略六：冷热分离

| 数据类型 | 存储位置          | 保留策略 | 说明               |
| -------- | ----------------- | -------- | ------------------ |
| 热数据   | Redis             | 最近7天  | 频繁查询，快速访问 |
| 温数据   | PostgreSQL        | 30天     | 偶尔查询，归档存储 |
| 冷数据   | PostgreSQL + 压缩 | 1年+     | 合规审计，长期保留 |

**优点**：平衡查询性能与存储成本
**缺点**：需要定时任务迁移数据
**适用场景**：需要长期追溯历史日志

#### 策略七：异步批量写入

```
日志产生 → 内存缓冲(批量) → 定时写入Redis → 失败重试/本地备份
```

| 参数     | 建议值     | 说明             |
| -------- | ---------- | ---------------- |
| 批量大小 | 100-1000条 | 按日志量调整     |
| 刷新间隔 | 1-5秒      | 平衡实时性与性能 |
| 重试次数 | 3次        | 失败后写入本地   |

**优点**：提高写入吞吐量，减少网络开销
**缺点**：存在秒级延迟，可能丢失缓冲区数据
**适用场景**：网络带宽有限、高并发写入

#### 策略八：多级缓冲

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  内存缓冲    │────▶│    Redis    │────▶│ PostgreSQL  │────▶│   归档存储   │
│  (秒级)     │     │   (天级)    │     │   (月级)    │     │   (年/永久)  │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

**优点**：高并发写入保障，数据可靠性递增
**缺点**：架构复杂度高
**适用场景**：高并发写入、要求高可靠性

#### 策略选择建议

**基础策略选择：**

| 场景       | 推荐策略 | 理由                   |
| ---------- | -------- | ---------------------- |
| 生产环境   | 策略一   | 平衡性能、成本、可靠性 |
| 开发测试   | 策略二   | 简单，日志量小         |
| 合规要求高 | 策略三   | 双重保障，满足审计     |
| 网络不稳定 | 策略四   | 减少网络依赖           |

**进阶策略选择：**

| 场景          | 推荐策略 | 理由                 |
| ------------- | -------- | -------------------- |
| 日志量巨大    | 策略五   | 采样降低存储成本     |
| 长期追溯      | 策略六   | 冷热分离，成本可控   |
| 高并发写入    | 策略七   | 批量写入提升吞吐     |
| 金融/核心系统 | 策略八   | 多级保障，最高可靠性 |

**组合策略建议：**

| 组合方案   | 策略组合        | 典型场景                |
| ---------- | --------------- | ----------------------- |
| 生产增强版 | 策略一 + 策略六 | 生产环境 + 长期追溯需求 |
| 边缘优化版 | 策略四 + 策略七 | 网络不稳定 + 高吞吐场景 |
| 成本优化版 | 策略一 + 策略五 | 日志量大 + 存储成本敏感 |
| 金融合规版 | 策略三 + 策略八 | 金融系统 + 审计合规要求 |

### 2.5 Go应用 - 日志采集与存储

极简版仅依赖 `go-redis`，输出Pino兼容格式：

```go
// pkg/logger/logger.go
package logger

import (
    "context"
    "encoding/json"
    "fmt"
    "os"
    "sync"
    "time"

    "github.com/redis/go-redis/v9"
)

// Pino日志级别常量
const (
    LevelTrace = 10
    LevelDebug = 20
    LevelInfo  = 30
    LevelWarn  = 40
    LevelError = 50
    LevelFatal = 60
)

// LogEntry 结构化日志 (Pino兼容格式)
type LogEntry struct {
    Name     string                 `json:"name"`           // 服务名称
    Hostname string                 `json:"hostname"`       // 节点ID
    Pid      int                    `json:"pid"`            // 进程ID
    Level    int                    `json:"level"`          // Pino日志级别
    Msg      string                 `json:"msg"`            // 消息
    Time     int64                  `json:"time"`           // 毫秒时间戳
    Svc      string                 `json:"svc"`            // 服务名
    Node     string                 `json:"node"`           // 节点ID
    Fields   map[string]interface{} `json:"fields,omitempty"`
}

// SimpleLogger 极简日志器
type SimpleLogger struct {
    redis     *redis.Client
    stream    string
    nodeID    string
    service   string
    localFile *os.File
    mu        sync.Mutex
}

// NewSimpleLogger 创建日志器
func NewSimpleLogger(redisAddr, nodeID, service string) (*SimpleLogger, error) {
    rdb := redis.NewClient(&redis.Options{
        Addr:     redisAddr,
        PoolSize: 5, // 边缘设备小连接池
    })

    // 测试连接
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    if err := rdb.Ping(ctx).Err(); err != nil {
        return nil, fmt.Errorf("连接Redis失败: %w", err)
    }

    // 本地文件作为备份
    logsDir := "logs"
    if err := os.MkdirAll(logsDir, 0755); err != nil {
        return nil, fmt.Errorf("创建日志目录失败: %w", err)
    }
    localPath := fmt.Sprintf("%s/%s-%s.log", logsDir, service, nodeID)
    f, err := os.OpenFile(localPath, os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
    if err != nil {
        return nil, fmt.Errorf("创建本地日志文件失败: %w", err)
    }

    return &SimpleLogger{
        redis:     rdb,
        stream:    "logs:stream",
        nodeID:    nodeID,
        service:   service,
        localFile: f,
    }, nil
}

// levelToPino 转换日志级别到Pino格式
func levelToPino(level string) int {
    switch level {
    case "trace":
        return LevelTrace
    case "debug":
        return LevelDebug
    case "info":
        return LevelInfo
    case "warn":
        return LevelWarn
    case "error":
        return LevelError
    case "fatal":
        return LevelFatal
    default:
        return LevelInfo
    }
}

// Log 记录日志 - 同时写入Redis和本地
func (l *SimpleLogger) Log(ctx context.Context, level, msg string, fields map[string]interface{}) {
    entry := LogEntry{
        Name:     l.service,
        Hostname: l.nodeID,
        Pid:      os.Getpid(),
        Level:    levelToPino(level),
        Msg:      msg,
        Time:     time.Now().UnixMilli(),
        Svc:      l.service,
        Node:     l.nodeID,
        Fields:   fields,
    }

    data, _ := json.Marshal(entry)

    // 1. 写入Redis (异步，失败不阻塞)
    go func() {
        l.redis.XAdd(ctx, &redis.XAddArgs{
            Stream: l.stream,
            MaxLen: 10000, // 保留最近1万条
            Values: map[string]interface{}{
                "data": string(data),
            },
        })
    }()

    // 2. 写入本地文件 (同步，确保不丢)
    l.mu.Lock()
    l.localFile.Write(append(data, '\n'))
    l.mu.Unlock()
}

// Info / Error / Warn 快捷方法
func (l *SimpleLogger) Info(ctx context.Context, msg string, fields map[string]interface{}) {
    l.Log(ctx, "info", msg, fields)
}

func (l *SimpleLogger) Error(ctx context.Context, msg string, fields map[string]interface{}) {
    l.Log(ctx, "error", msg, fields)
}

func (l *SimpleLogger) Warn(ctx context.Context, msg string, fields map[string]interface{}) {
    l.Log(ctx, "warn", msg, fields)
}
```

```go
// pino/reader.go
package pino

import (
    "context"
    "fmt"
    "os/exec"

    "github.com/redis/go-redis/v9"
)

// PinoReader 读取Redis日志并输出给Pino
type PinoReader struct {
    redis  *redis.Client
    stream string
}

// Tail 实时跟踪日志 (类似 tail -f)
func (pr *PinoReader) Tail(ctx context.Context) {
    lastID := "$"
  
    for {
        result, err := pr.redis.XRead(ctx, &redis.XReadArgs{
            Streams: []string{pr.stream, lastID},
            Count:   100,
            Block:   0, // 阻塞等待
        }).Result()
  
        if err != nil {
            continue
        }
  
        for _, stream := range result {
            for _, msg := range stream.Messages {
                // 直接输出JSON，Pino可以解析
                fmt.Println(msg.Values["data"])
                lastID = msg.ID
            }
        }
    }
}

// PipeToPino 管道到pino-pretty命令
func (pr *PinoReader) PipeToPino(ctx context.Context) error {
    cmd := exec.CommandContext(ctx, "npx", "pino-pretty", "-t")
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
  
    stdin, err := cmd.StdinPipe()
    if err != nil {
        return err
    }
  
    if err := cmd.Start(); err != nil {
        return err
    }
  
    // 读取Redis并写入stdin
    lastID := "0"
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            result, _ := pr.redis.XRead(ctx, &redis.XReadArgs{
                Streams: []string{pr.stream, lastID},
                Count:   100,
                Block:   1000,
            }).Result()
  
            for _, stream := range result {
                for _, msg := range stream.Messages {
                    stdin.Write([]byte(msg.Values["data"].(string) + "\n"))
                    lastID = msg.ID
                }
            }
        }
    }
}
```

```go
// analyzer/simple_analyzer.go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "os"
    "strings"
    "time"

    "github.com/redis/go-redis/v9"
    "github.com/sashabaranov/go-openai"
)

// SimpleAnalyzer 极简分析器
type SimpleAnalyzer struct {
    llm    *openai.Client
    redis  *redis.Client
    stream string
}

// NewSimpleAnalyzer 创建分析器
func NewSimpleAnalyzer(apiKey, redisAddr string) *SimpleAnalyzer {
    return &SimpleAnalyzer{
        llm:    openai.NewClient(apiKey),
        redis:  redis.NewClient(&redis.Options{Addr: redisAddr}),
        stream: "logs:stream",
    }
}

// AnalyzeRecent 分析最近N条日志
func (sa *SimpleAnalyzer) AnalyzeRecent(ctx context.Context, count int64) (string, error) {
    // 从Redis读取日志
    msgs, err := sa.redis.XRevRangeN(ctx, sa.stream, "+", "-", count).Result()
    if err != nil {
        return "", err
    }

    // 构建日志文本
    var logs []string
    for _, msg := range msgs {
        logs = append(logs, msg.Values["data"].(string))
    }

    // 调用LLM分析
    return sa.callLLM(ctx, logs)
}

// callLLM 调用大模型
func (sa *SimpleAnalyzer) callLLM(ctx context.Context, logs []string) (string, error) {
    prompt := fmt.Sprintf(`分析以下边缘设备日志，给出异常检测和根因分析：

%s

请用中文简要回答：
1. 是否有异常？
2. 异常类型和可能原因
3. 建议措施`, strings.Join(logs, "\n"))

    resp, err := sa.llm.CreateChatCompletion(ctx, openai.ChatCompletionRequest{
        Model: "gpt-3.5-turbo", // 使用轻量级模型降低成本
        Messages: []openai.ChatCompletionMessage{
            {Role: "system", Content: "你是边缘设备运维助手，擅长日志分析。"},
            {Role: "user", Content: prompt},
        },
        MaxTokens: 500, // 限制输出长度
    })
    if err != nil {
        return "", err
    }

    return resp.Choices[0].Message.Content, nil
}

// WatchAndAnalyze 持续监控并分析
func (sa *SimpleAnalyzer) WatchAndAnalyze(ctx context.Context, interval time.Duration) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()

    for {
        select {
        case <-ticker.C:
            result, err := sa.AnalyzeRecent(ctx, 50)
            if err == nil && strings.Contains(result, "异常") {
                fmt.Printf("[ALERT] %s\n%s\n\n", time.Now().Format("15:04:05"), result)
            }
        case <-ctx.Done():
            return
        }
    }
}
```

## 六、PostgreSQL 持久化存储

### 6.1 数据库连接配置

```go
// pkg/storage/postgres.go
package storage

import (
    "database/sql"
    "fmt"
    "time"

    _ "github.com/lib/pq"
)

// PostgresConfig PostgreSQL配置
type PostgresConfig struct {
    Host     string
    Port     int
    User     string
    Password string
    Database string
}

// PostgresStorage PostgreSQL存储
type PostgresStorage struct {
    db *sql.DB
}

// NewPostgresStorage 创建PostgreSQL存储
func NewPostgresStorage(cfg PostgresConfig) (*PostgresStorage, error) {
    // 首先连接到默认的postgres数据库，检查并创建目标数据库
    defaultDSN := fmt.Sprintf("host=%s port=%d user=%s password=%s dbname=postgres sslmode=disable",
        cfg.Host, cfg.Port, cfg.User, cfg.Password)
  
    defaultDB, err := sql.Open("postgres", defaultDSN)
    if err != nil {
        return nil, fmt.Errorf("连接PostgreSQL失败: %w", err)
    }
    defer defaultDB.Close()

    // 检查数据库是否存在
    var exists bool
    err = defaultDB.QueryRow("SELECT EXISTS(SELECT 1 FROM pg_database WHERE datname = $1)", cfg.Database).Scan(&exists)
    if err != nil {
        return nil, fmt.Errorf("检查数据库失败: %w", err)
    }

    // 如果数据库不存在，创建它
    if !exists {
        _, err = defaultDB.Exec(fmt.Sprintf("CREATE DATABASE %s", cfg.Database))
        if err != nil {
            return nil, fmt.Errorf("创建数据库失败: %w", err)
        }
    }

    // 连接到目标数据库
    dsn := fmt.Sprintf("host=%s port=%d user=%s password=%s dbname=%s sslmode=disable",
        cfg.Host, cfg.Port, cfg.User, cfg.Password, cfg.Database)
  
    db, err := sql.Open("postgres", dsn)
    if err != nil {
        return nil, err
    }
  
    // 配置连接池
    db.SetMaxOpenConns(10)
    db.SetMaxIdleConns(5)
    db.SetConnMaxLifetime(time.Hour)
  
    if err := db.Ping(); err != nil {
        return nil, err
    }
  
    return &PostgresStorage{db: db}, nil
}
```

## 七、第三方日志文件采集

### 7.1 问题背景

### 7.2 file-collector 功能特性

| 功能         | 说明                                    |
| ------------ | --------------------------------------- |
| 文件监控     | 使用fsnotify实时监控日志文件变化        |
| 增量读取     | 记录读取位置，只读取新增内容            |
| 日志轮转检测 | 自动检测文件重命名/重建，从头读取新文件 |
| 多格式解析   | 支持Log4j、JSON、Syslog等多种格式       |
| 自动格式检测 | 智能识别日志格式，无需手动配置          |

### 7.3 支持的日志格式

**Log4j/Logback格式:**

```
[INFO] 2024-03-24 15:30:45,123 [main] com.example.App - Application started
[ERROR] 2024-03-24 15:30:48,456 [http-handler] com.example.Handler - Connection timeout
```

**JSON格式:**

```json
{"level":30,"time":"2024-03-24T15:30:45.123Z","msg":"Application started","svc":"app"}
```

**Syslog格式:**

```
Mar 24 15:30:45 hostname process[pid]: message
```

### 7.4 使用方式

```bash

# 指定日志格式
./bin/file-collector.exe -files logs/app.log -format log4j -service my-app

# 完整参数

  -redis localhost:6379 \
  -node edge-collector \
  -service third-party-app \
  -format auto \
```

### 7.5 参数说明

| 参数     | 默认值           | 说明                                    |
| -------- | ---------------- | --------------------------------------- |

### 7.6 日志轮转处理

| 场景           | 检测方式                   | 处理方法                  |

| 文件被删除重建 | 文件不存在后重新出现       | 从头读取新文件            |

## 十、后续优化设想

以下为方案层面的优化建议，当前演示程序暂不实现，仅供后续演进参考。

### 10.1 分层分析架构

当前系统采用"规则引擎 + LLM"的两层分析模式，可考虑扩展为三层架构：

```
边缘节点 → 规则引擎(边缘) → LLM分析(云端) → 人工复核(可选)
     ↓           ↓              ↓
  本地缓存    初步告警       根因分析
```

**优化要点**：

- 边缘侧仅做轻量级规则匹配和关键告警
- 复杂分析任务交给云端 LLM 处理
- 支持人工复核形成闭环

### 10.2 数据分级策略

不是所有日志都需要上报云端，建议采用分级处理机制：

| 级别     | 触发条件             | 处理方式            |
| -------- | -------------------- | ------------------- |
| Critical | ERROR + 关键规则匹配 | 立即上报 + 实时告警 |
| Warning  | WARN 或频繁 ERROR    | 本地缓存 + 批量上报 |
| Info     | 普通运行日志         | 本地存储，定期上报  |

**优化要点**：

- 减少无效数据传输，降低网络带宽压力
- 关键告警优先处理，确保问题及时响应
- 不同级别采用不同的存储策略，平衡成本与追溯能力

### 10.3 离线容灾机制

边缘节点可能面临网络中断的情况，需要增强离线能力：

**优化要点**：

- **本地日志缓冲**：Redis 不可用时，写入本地文件队列（如 SQLite 或独立日志文件）
- **断点续传**：网络恢复后，自动同步积压数据
- **数据去重**：基于时间戳 + 节点ID + 日志hash，避免重复上报

### 10.4 多通道上报

当前依赖单一 Redis 通道，可增加备选上报通道以提高可靠性：

| 通道         | 优先级 | 适用场景                 |
| ------------ | ------ | ------------------------ |
| Redis Stream | 主通道 | 低延迟、高吞吐的内部网络 |
| HTTP POST    | 备选1  | 云端 API 接入            |
| MQTT         | 备选2  | 需要公网传输的场景       |

**优化要点**：

- 主通道失败时自动切换到备选通道
- 支持配置多通道同时写入，实现多活

### 10.5 安全与认证

**节点身份认证**：

- 节点使用预置证书或密钥
- 每条日志携带 token 或签名
- 云端验证后才入库

**数据传输加密**：

- 传输层：TLS/HTTPS 加密
- 应用层：敏感日志内容可选 AES 加密

### 10.6 分布式追踪

当前日志分散在各个边缘节点，建议实现跨节点追踪：

**优化要点**：

- 每条日志增加 trace_id 字段
- 关联同一请求在不同服务间的日志
- 便于问题定位和根因分析

### 10.7 指标采集与监控

增加运行时指标采集：

| 指标         | 说明            |
| ------------ | --------------- |
| 日志吞吐量   | 条/秒           |
| 规则匹配率   | 匹配数/总日志数 |
| LLM 分析延迟 | 分析耗时        |
| 队列积压长度 | 待处理日志数量  |

**优化要点**：

- 支持 Prometheus 格式导出
- 可接入现有运维监控系统

### 10.8 规则版本管理

**优化要点**：

- 规则版本号 + 变更记录
- 新规则先在测试节点验证
- 支持快速回滚

### 10.9 自动告警收敛

避免告警风暴，提高运维效率：

| 机制     | 说明                          |
| -------- | ----------------------------- |
| 告警合并 | 同类告警在 N 分钟内只发送一次 |
| 告警升级 | 未响应则自动升级通知级别      |
| 告警静默 | 维护窗口期静默通知            |

### 10.10 根因分析知识库

结合知识库实现更智能的分析：

- 常见错误码 → 知识库映射（自动给出初步建议）
- 相似历史案例推荐（类似问题的处理记录）
- 自动化故障自愈建议（基于历史经验生成修复方案）

### 10.11 多租户支持

如需面向多个客户/业务线：

- 租户隔离（逻辑隔离或物理隔离）
- 租户专属规则集配置
- 数据权限控制（按租户过滤查询结果）

## 九、总结

本极简方案的核心组件：

1. **Go应用** - 内置日志采集，直接写入Redis
2. **file-collector** - 采集第三方程序日志文件
3. **Redis** - 同时作为存储和消息队列
4. **Pino** - 超高性能日志管理CLI工具，比Bunyan快5倍
5. **Web 可视化界面** - 基于 Fastify + Pino 的日志查看器，支持实时日志流、规则分析、LLM智能分析
6. **大模型** - 按需调用，无常驻服务
7. **PostgreSQL** - 分析结果持久化存储

**核心优势：**

- **极简**：仅依赖 Redis + PostgreSQL，无 Kafka/ES 等重量级组件
- **轻量**：占用资源显著少于ELK方案
- **高性能**：Pino提供业界最快的JSON日志处理能力
- **易用**：Pino-pretty 提供强大的日志查询和格式化能力；Web界面提供可视化操作
- **无侵入**：file-collector 无需修改第三方程序即可采集日志
- **智能**：大模型按需分析，降低成本
- **持久化**：PostgreSQL 存储分析结果，支持历史查询和趋势分析
- **可视化**：Web界面提供日志列表、实时流、详情查看、规则分析和LLM智能分析功能
