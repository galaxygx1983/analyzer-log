## 概述

### Q1. 本文档的目的是什么？

本文档定义融合平台与智能视频分析系统之间的API接口标准，作为双方技术团队进行系统集成开发的唯一依据和事实来源。

*来源: 1 概述*

### Q2. 智能视频分析系统负责什么功能？

智能视频分析系统负责对指定的摄像头视频码流进行实时AI推理分析。融合平台通过调用本文档定义的接口，获取其关注的分析结果，主要包括实时告警信息与客流统计数据。

*来源: 1 概述*

### Q3. 系统间的交互模式有哪两种？

系统间的交互模式分为两种：
- **数据推送 (Push)**: 由智能视频分析系统作为客户端，主动将实时告警和实时客流数据推送至融合平台提供的API接口。
- **数据查询 (Pull)**: 由融合平台作为客户端，主动调用智能视频分析系统提供的API接口，查询历史客流等数据。

*来源: 1 概述*

## 实时告警推送接口

### Q1. 实时告警推送接口的基本信息是什么？

- **接口类型**: 推送 (Push)
- **接口提供方**: 融合平台
- **接口调用方**: 智能视频分析系统
- **URL**: `/CQlineTF/service/S_SP_01`
- **Method**: POST
- **Content-Type**: application/json
- **认证**: 无

*来源: 2.1-2.2 接口定义*

### Q2. 实时告警推送接口的请求参数有哪些？

| 参数 | 类型 | 说明 |
|-----|------|------|
| alertType | String | 告警类型编码，必须与告警类型编码表中定义的值一致 |
| cameraNationalCode | String | 20位摄像头国标码 |
| hikPlanID | String | 5位海康平台规划ID |
| createTime | String | 告警事件发生的时间，格式为 YYYY-MM-DD HH:mm:ss |
| file | String | 指向告警证据文件（图片或视频）的可访问URL |

*来源: 2.3 请求体*

### Q3. 实时告警推送接口的响应逻辑是什么？

此接口的成功与否判断比较特殊。无论业务处理是否成功，只要HTTP通信正常，HTTP状态码总是 200 OK。真正的处理结果需要通过解析响应体中的 `respCode` 字段来确定：
- 成功：`respCode` 的值为 `"0"`
- 业务异常：`respCode` 的值为非 `"0"` 的字符串

*来源: 2.5 响应说明*

### Q4. 实时告警推送接口的部署说明是什么？

融合平台的告警接收服务是按站点独立部署的。智能视频分析系统在推送告警时，必须根据告警所属站点，将请求发送到该站点对应的URL。例如，A站的告警需推送到 `http://{A站IP}:{端口}/CQlineTF/service/S_SP_01`。

*来源: 2.2 接口定义*

### Q5. 实时告警推送接口的请求示例是什么？

```json
{
    "alertType": "STANDING_WATER",
    "cameraNationalCode": "50000000001310353001",
    "hikPlanID": "10003",
    "createTime": "2025-08-27 14:47:45",
    "file": "http://10.247.84.163:8080/u/A/20250827/001207852804.jpg"
}
```

*来源: 2.4 请求示例*

### Q6. 当告警推送失败时应如何处理？

如果 `respCode` 非 "0"，或 HTTP 状态码非 200，或发生网络超时，都应视为失败。智能视频分析系统应将此次推送标记为失败，并在后续的推送周期中进行重试。

*来源: 2.5 响应说明*

## 实时客流数据推送接口

### Q1. 实时客流数据推送接口的基本信息是什么？

- **接口类型**: 推送 (Push)
- **接口提供方**: 融合平台
- **接口调用方**: 智能视频分析系统
- **URL**: `/CQlineTF/service/S_SP_02`
- **Method**: POST
- **Content-Type**: application/json
- **认证**: 无

*来源: 3.1-3.2 接口定义*

### Q2. 实时客流数据推送接口的推送周期是什么？

智能视频分析系统根据可配置的周期（默认为1-5分钟）调用此接口，上报客流分析数据。

*来源: 3.1 概述*

### Q3. 实时客流数据推送接口的请求参数有哪些？

| 参数 | 类型 | 说明 |
|-----|------|------|
| stationName | String | 站点名称，必须与站点位置编码中定义的官方中文名称一致 |
| detectType | String | 检测类型编码，例如 "HorizontalCrowdCount" |
| createTime | String | 数据生成时间，格式为 YYYY-MM-DD HH:mm:ss |
| crowdCountList | Array | 按摄像头统计的人数列表，每个对象包含 cameraNationalCode、hikPlanID 和 crowdCount |
| countTypeList | Array | 按区域统计的人数列表，每个对象包含 countType（20位编码）、countTypeDesc 和 crowdCount |

*来源: 3.3 请求体*

### Q4. 实时客流数据推送接口的请求示例是什么？

```json
{
    "stationName": "商贸城站",
    "detectType": "HorizontalCrowdCount",
    "createTime": "2025-08-27 15:09:11",
    "crowdCountList": [
        {"cameraNationalCode": "50000000001310847001", "hikPlanID": "10005", "crowdCount": "30"},
        {"cameraNationalCode": "50000000001310847000", "hikPlanID": "10004", "crowdCount": "70"}
    ],
    "countTypeList": [
        {"countType": "50010811240080200101", "countTypeDesc": "商贸城站-站厅-区域1", "crowdCount": "30"},
        {"countType": "50010811240080100101", "countTypeDesc": "商贸城站-站台-区域1", "crowdCount": "70"}
    ]
}
```

*来源: 3.4 请求示例*

### Q5. 实时客流数据的统计结果说明是什么？

统计人数为AI推理结果，可能存在一定偏差。推送内容包含基于AI算法估算的各监控区域内的人数，以及推测的车站总人数。

*来源: 3.1 概述*

## 历史客流数据查询接口

### Q1. 历史客流数据查询接口的基本信息是什么？

- **接口类型**: 查询 (Pull)
- **接口提供方**: 智能视频分析系统
- **接口调用方**: 融合平台
- **URL**: `/api/v1/analytics/crowd_history`
- **Method**: POST
- **Content-Type**: application/json
- **认证**: 无

*来源: 4.1-4.2 接口定义*

### Q2. 历史客流数据查询接口的用途是什么？

该接口采用现代化RESTful风格设计，为融合平台提供一个高效、标准的历史客流数据查询方式，主要用于客流密度热力图等场景。接口返回指定时间范围内，按分析区域/分组聚合后的客流总数。

*来源: 4.1 概述*

### Q3. 历史客流数据查询接口的请求参数有哪些？

| 参数 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| station_names | Array of Strings | 否 | 要查询的站点官方中文名列表，省略或为空数组则查询所有站点 |
| start_time | String | 否 | 查询起始时间，ISO 8601格式的UTC时间，省略则默认为30天前 |
| end_time | String | 否 | 查询结束时间，ISO 8601格式，省略则默认为当前时间 |

*来源: 4.3 请求体*

### Q4. 历史客流数据查询接口的数据保留期限是什么？

智能视频分析系统仅保留最近30天的数据，超出此范围的查询请求将无法返回结果。

*来源: 4.3 请求体*

### Q5. 历史客流数据查询接口的响应格式是什么？

成功响应（HTTP 200 OK）返回JSON对象，包含：
- `response_timestamp`: 响应时间戳
- `query_parameters`: 查询参数回显
- `data`: 按站点分组的查询结果数组，每个站点包含 `station_name` 和 `results`（区域编码、描述、总人数）

错误响应包括：
- 422 Unprocessable Entity：参数格式错误
- 500 Internal Server Error：服务端内部错误

*来源: 4.5 响应说明*

## 数据约定

### Q1. 字符编码要求是什么？

为确保跨平台兼容性（如 Windows, Linux），所有接口中包含非ASCII字符的字符串（例如 stationName, countTypeDesc, station_names 等），都必须使用 UTF-8 编码。

*来源: 5.1 概述*

### Q2. 告警类型编码有哪些？

| 功能名称 | 任务类型编码 | 告警类型编码 (alertType) |
|---------|-------------|------------------------|
| 围界闯入 | borderDetect | BORDER |
| 人员摔倒 | fallDownDetect | FALLDOWN |
| 扶梯逆行 | reverseDetect | REVERSE |
| 大体积物品 | futiDetect | BIGSIZE_OBJECT |
| 异常行李 | futiDetect | ABORMAL_OBJECT |
| 人员滞留 | LoiteringDetect | LOITERING |
| 客流密度异常 | HorizontalCrowdCount | HEAVY_CROWD |
| 隧道积水 | StandingWaterDetect | STANDING_WATER |

*来源: 5.2 告警类型编码*

### Q3. 摄像头编码有哪两种？

为兼容融合平台内部不同的系统，API将同时提供两种摄像头标识符：
- **cameraNationalCode**: 20位国标编码 (GB/T 28181)
- **hikPlanID**: 5位海康平台规划ID，保证全线唯一

*来源: 5.3 摄像头编码*

### Q4. 分析区域与分组编码的结构是什么？

20位编码结构：
| 位码 | 含义 | 长度 | 说明 |
|-----|------|------|------|
| 1-6 | 行政区划代码 | 6 | 国标 GB/T 2260，例：重庆市南岸区为 500108 |
| 7-8 | 项目类型 | 2 | 例：11=地铁交通, 12=机场, 13=智慧园区 |
| 9-10 | 线路/项目编号 | 2 | 例：地铁 24 号线 |
| 11-13 | 站点/位置编号 | 3 | 线路内的具体位置 |
| 14-15 | 主功能区 | 2 | 例：00=全站, 01=站台, 02=站厅 |
| 16-18 | 子区域/分组序号 | 3 | 主功能区内自定义区域的流水号 |
| 19-20 | 区域类型 | 2 | 例：01=物理区域, 02=逻辑分组 |

*来源: 5.5 分析区域与分组编码*
